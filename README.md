# VARTA

Система оповіщення пішоходів про небезпечні транспортні засоби в реальному часі.

Хакатон «Місто майбутнього» · Команда TechSisters

## Що це за проєкт

Транспорт (велосипед, самокат) постійно надсилає координати й швидкість на сервер. Сервер обчислює, чи становить цей транспорт загрозу комусь із пішоходів поблизу, і якщо так — миттєво надсилає сповіщення на пристрій пішохода (вібрація / червоний екран).

MVP не використовує реальне залізо — транспорт і пішохід емулюються скриптом і веб-сторінкою.

## Архітектура (коротко)

| Компонент | Технологія | Що робить |
|---|---|---|
| Ingestion API | FastAPI, `POST /api/telemetry` | Приймає координати від транспорту/пішохода, кладе у словник |
| In-memory state | `dict` + `asyncio.Lock` | "Гарячі" дані — поточні координати й швидкість усіх пристроїв |
| Risk Engine | фоновий asyncio-цикл | Раз на 0.5–1 с перевіряє дистанції/вектори, шукає загрози |
| Сповіщення | FastAPI WebSocket `/ws/{client_id}` | Миттєво пушить сигнал конкретному пішоходу |
| Реєстр (холодні дані) | PostgreSQL | Зареєстровані пристрої, користувачі, налаштування — поки не підключено |

Чому WebSocket, а не звичайні запити з телефону кожну секунду (polling), і чому одна асинхронна функція замість потоків/процесів — детальніше можна запитати в чаті, тут не дублюю, щоб README не розростався.

## Структура проєкту

```text
.
├── main.py              # точка входу, FastAPI-застосунок
├── schemas.py           # Схема валідації (Pydantic)
├── seed.py              # наповнення бази тестовими даними
├── db.py                # налаштування підключення до PostgreSQL
├── models.py            # SQLAlchemy схеми таблиць бази даних
├── docker-compose.yml   # конфігурація для швидкого підняття БД
├── requirements.txt     # залежності
├── .env.example         # шаблон змінних середовища (без реальних секретів)
├── SETUP.md              # детальна інструкція встановлення (якщо щось не запускається)
└── .gitignore
```

Поки весь код в `main.py` — це тимчасово, для швидкої перевірки MVP. Далі плануємо розбити на `state.py`, `risk_engine.py`, `connection_manager.py`, `db.py`.

## Швидкий старт

Спершу потрібно підняти базу даних (переконайся, що Docker запущений на твоєму комп'ютері):

```bash
docker-compose up -d
```

Потім класичний запуск бекенду:

```bash
python3 -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
uvicorn main:app --reload --port 8000
```

Перевірка: відкрити `http://127.0.0.1:8000/docs` (Swagger UI), там можна потестити `POST /api/telemetry`.
Для перевірки підключення до БД GET /api/health.

Якщо щось не запускається або потрібні деталі по кожному кроку — дивись `SETUP.md`.

## Емулятор (Тестування в реальному часі)

Щоб не надсилати координати вручну через Swagger, у проєкті є готовий скрипт `emulator.py`. Він повністю імітує реальну поведінку системи:
1. Підключається до веб-сокета від імені пішохода (`pedestrian_1`), щоб слухати ефір.
2. Щосекунди відправляє телеметрію (POST-запити) для пішохода та швидкохідного самоката (`scooter_1`), рухаючи їх назустріч одне одному.

### Як запустити:
1. Переконайся, що база даних та основний бекенд **вже запущені** (через `docker-compose up -d` та `uvicorn main:app --reload`). 
   *Примітка: база має бути автоматично наповнена тестовими пристроями при старті сервера (сідінг).*
2. Відкрий **нове вікно терміналу** (щоб не зупиняти сервер).
3. Активуй віртуальне середовище:
   * Windows: `venv\Scripts\activate`
   * Mac/Linux: `source venv/bin/activate`
4. Запусти скрипт:
   ```bash
   python emulator.py

### `WS /ws/{client_id}`

WebSocket-з'єднання для пішохода. Поки відкрите — на нього прилітають сповіщення про небезпеку. У Swagger не тестується (Swagger не вміє WebSocket), потрібен окремий клієнт.
